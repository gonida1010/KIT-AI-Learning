# AI 활용 전략

> Edu-Sync AI 프로젝트에서 AI 도구를 어떻게 활용했는지에 대한 전략 문서

---

## 1. 사용한 AI 도구 및 선택 이유

### 1-1. GitHub Copilot (Claude Opus 4) — 개발 도구

| 항목 | 내용 |
| ---- | ---- |
| **용도** | 프로젝트 전체 코드 작성, 디버깅, 리팩터링 |
| **선택 이유** | Agent 모드에서 프로젝트 폴더 전체를 컨텍스트로 잡고 작업 가능 |
| **핵심 장점** | 파일 간 연결이 끊어지지 않고 일관된 코드 생성, 여러 파일 동시 수정 |

**활용 방식:**
- Agent 모드로 "이런 기능 만들어줘" 지시 → 백엔드 API + 프론트엔드 UI + DB 모델 동시 수정
- 코드 생성 후 자동 오류 검사 → 문제 발견 시 자체 수정 루프
- 보안 사항(API 키 환경변수 분리, 입력 검증, 인증 토큰 확인) 자동 적용

### 1-2. OpenAI GPT-4o-mini (LangChain) — 서비스 내장 AI

| 항목 | 내용 |
| ---- | ---- |
| **용도** | 챗봇 응답, 큐레이션 요약, 브리핑 리포트 생성, 스케줄 파싱 |
| **선택 이유** | 비용 대비 한국어 성능 우수, 빠른 응답 속도 |
| **핵심 장점** | 학원 서비스 특성상 한국어 이해·생성 품질이 높아야 하며, 학생 수십 명의 일일 호출 비용 통제 가능 |

**활용 방식:**
- 모든 프롬프트를 **영어로 작성**하고 출력만 한국어로 지시 → 이중 번역 오버헤드 제거, 응답 속도 개선
- JSON 형식 응답으로 고정 → 파싱 안정성 확보, 불필요한 토큰 절감
- LangChain 비동기 호출로 동시 요청 처리

### 1-3. FAISS (벡터 검색 엔진) — RAG 인프라

| 항목 | 내용 |
| ---- | ---- |
| **용도** | 지식 베이스, 큐레이션, 멘토 자료 의미 기반 검색 |
| **선택 이유** | 별도 서버 없이 파일 형태 동작, 학원 규모(수백~수천 건)에 충분한 성능 |
| **핵심 장점** | 단순 키워드 매칭이 아닌 의미적 유사도 기반 검색 |

**활용 방식:**
- 3개 독립 벡터스토어: 공통 지식 / 큐레이션 / 멘토별 자료
- 멘토별 자료는 **최신 자료·기초 자료 분리** 저장, 학생의 담당 멘토 자료만 검색
- 문서 삭제 시 벡터스토어 자동 재구축 (동기화)

## 2. 멀티 에이전트 AI 아키텍처

```
학생 메시지 입력
        ↓
┌─────────────────────┐
│   Router AI         │  ← 의도 분류 (행정/학습/상담)
│   (GPT-4o-mini)     │
└────────┬────────────┘
         ├── category: admin/career/curation
         │   └── [Agent A] RAG 검색 + 답변 생성
         │       ├── 공통 지식 벡터스토어 검색
         │       ├── 큐레이션 벡터스토어 검색
         │       └── 멘토 자료 벡터스토어 검색
         │
         ├── category: study/booking
         │   └── [Agent B] 학습 질문 답변 + 예약 플로우
         │       ├── 예약 가능 시간 실시간 조회
         │       ├── 4단계 예약 프로세스
         │       └── AI 브리핑 리포트 자동 생성
         │
         └── category: handoff
             └── [Human Handoff] 멘토 1:1 알림
                 └── 멘토 대시보드에 상담 요청 표시
```

### 에이전트별 프롬프트 전략

| 에이전트 | 프롬프트 언어 | 출력 언어 | JSON 형식 강제 | 비고 |
| -------- | ------------- | --------- | -------------- | ---- |
| Router   | 영어          | 영어(분류값) | ✅ | 분류만 수행, 생성 없음 |
| Agent A  | 영어          | 한국어    | ✅ | RAG 컨텍스트 주입, 자료 카드 포함 |
| Agent B  | 영어          | 한국어    | ✅ | 예약 상태 관리, 브리핑 생성 |
| 큐레이션 요약 | 영어     | 한국어    | ✅ | 파일 업로드 시 자동 실행 |
| 스케줄 파싱   | 영어     | 한국어    | ✅ | 자연어 → 날짜/시간 구조 변환 |

## 3. 토큰 절약 전략

| 전략 | 설명 |
| ---- | ---- |
| **GPT-4o-mini 선택** | Full 모델 대비 비용 90%+ 절감, 학원 서비스에 충분한 품질 |
| **영어 프롬프트** | 한국어 프롬프트 대비 토큰 수 감소 + 이중 번역 오버헤드 제거 |
| **JSON 응답 고정** | 자유 텍스트 대비 불필요한 토큰 생성 방지 |
| **검색 결과 제한** | 벡터 검색 상위 4개만 컨텍스트에 주입 |
| **텍스트 길이 제한** | 큐레이션 원문 400자, 카카오톡 응답 990자 제한 |

## 4. 유지보수성 전략

| 전략 | 설명 |
| ---- | ---- |
| **환경변수 기반 모델 전환** | `OPENAI_MODEL` 변수 하나로 모델 교체 가능 |
| **LLMProvider 추상화** | AI 모델이 바뀌어도 서비스 코드 수정 불필요 |
| **에이전트별 프롬프트 분리** | 각 에이전트의 동작 방식 변경 시 해당 파일만 수정 |
| **벡터스토어 자동 재구축** | 문서 추가/삭제 시 FAISS 인덱스 자동 동기화 |

## 6. AI 협업 과정 요약

이 프로젝트는 기획부터 배포까지 **GitHub Copilot (Claude Opus 4)**과 협업하여 개발했습니다.

### 주요 협업 이력

| 단계 | 작업 내용 | AI 활용 방식 |
| ---- | --------- | ------------ |
| 기획 | 프로젝트 구조 설계, 기술 스택 선정 | AI와 대화하며 아키텍처 결정 |
| 백엔드 | FastAPI 라우터, 서비스, DB 모델 구현 | Agent 모드로 다중 파일 동시 생성 |
| 프론트엔드 | React 페이지, 컴포넌트, 상태 관리 | UI 디자인 지시 → 코드 자동 생성 |
| RAG | FAISS 벡터스토어 구축, 검색 로직 | 전체 RAG 파이프라인 구현 |
| 카카오 연동 | 카카오톡 채널 웹훅 통합 | API 연동 코드 생성 |
| 배포 | Render 배포, PostgreSQL 마이그레이션 | 배포 설정 파일 생성, 마이그레이션 스크립트 |
| 버그 수정 | 실시간 테스트 → 버그 리포트 → 즉시 수정 | 오류 분석 + 코드 수정 + 배포 자동화 |
| 성능 최적화 | 프롬프트 영어 전환, 모델 교체 | 성능 병목 분석 + 프롬프트 리팩터링 |
